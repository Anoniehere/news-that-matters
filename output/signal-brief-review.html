Step 2 — Event Detection (Neural AI Clustering)

The system's pattern-recognition brain — understanding which articles are about the same real-world event

✅ Operational
What it does (plain English)

The AI reads every article and converts it into a mathematical "fingerprint" that captures its meaning — not just its words. Articles with similar fingerprints get grouped together as one event. Each group = one story the world is paying attention to.

Step 4 — LLM Enrichment

The insight layer — where raw trending events become actionable intelligence

▶ M3 — Next

For each of the top 5 ranked events, an LLM reads all source articles and produces four structured outputs. This is where the product stops being a news aggregator and becomes a thinking partner.

📌 Event Heading
≤ 15 words. The thesis sentence of the event.
📝 Summary
4–8 lines. What happened, grounded in source articles only. No invented facts.
🎯 Why It Matters
4–8 lines. Causal reasoning, written for a 35-year-old Silicon Valley PM/investor.
🏭 Sectors Impacted
1–5 sectors, ranked by confidence. E.g. "Tech: 0.92, Finance: 0.74"
Guardrails: System prompt explicitly bans financial advice, price targets, and buy/sell signals. Pydantic schema validation enforces output structure. Retries up to 2× on schema failure. LLM only uses facts from source articles — no invented context.

The Embedding Architecture — Why It's Two-Tier

The system has two modes for "reading" articles, with automatic failover. Think of it as having a PhD linguist on standby, with a smart keyword-matching assistant as backup.

🧠
Mode 1 — Neural (active now)

87MB neural language model runs locally on your GPU. Understands context and meaning — not just word overlap. "Fed" and "Federal Reserve" are the same thing. "Hike rates" and "raise benchmark" are the same action. Used on all production runs.

📝
Mode 2 — TF-IDF (safety fallback)

Word-frequency matching. Fast, zero setup, always works. Miss rate ~40% higher than neural — same event with different vocabulary often won't cluster. Only activates if the neural model file is missing. Never used in practice once model is downloaded.

Social Signal — Two-Tier Architecture (ADR-014)

Tier 1: Google Trends (pytrends) Currently blocked

Would give us normalized search interest score 0–100 for each event's topic, representing real public attention. Currently returning HTTP 400 due to a Google API auth change in late 2024 (affects pytrends library globally, not just Walmart network). Will auto-activate in production deployment.

signal_source: "pytrends" → score: [0.0–1.0]
Tier 2: Feed Diversity (active now) Running

Measures how many of our 5 editorial RSS feeds independently covered the same event. A story in Economy + US Politics + Top Stories = three independent editorial decisions = real signal. Completely offline, 100% reproducible, never fails.

signal_source: "feed_diversity" → (unique_feeds-1)/(5-1)
Why feed diversity is actually a better proxy than you might think: Each of our 5 feeds has a dedicated editorial team making independent curation decisions. When the "Economy" feed, the "US Politics" feed, and the "AI & Science" feed all independently surface the same story — that's three editorial teams agreeing. That signal is arguably more trustworthy than Google search volume, which can be gamed and includes noise from curiosity searches vs. genuine news importance.

Architectural Decision Log

Every significant intelligence or product trade-off made, with rationale. These shape what the system is and isn't.

ADR-003

X + LinkedIn dropped → Google Trends as social proxy

Accepted
WHY

X (Twitter) API starts at $100/month with severe rate limits. LinkedIn has no public content search API — requires a formal partner program application. Neither is viable for an MVP. Google Trends is free and covers the same real-world attention signal.

IMPACT

Trend scoring lost two social signals. Weights redistributed: repetition 70%, Google Trends 30%. Trend scores are slightly less socially-validated but far more economically rational for an MVP. V1.1 can re-add Reddit (PRAW, free) as a signal.

ADR-010

Reddit PRAW dropped for MVP — pure rep + Google Trends

Accepted
WHY

Reddit PRAW API (free tier) would require building subreddit relevance mapping, upvote normalization, and spam filtering. Estimated 1.5 engineering days to do properly. Not worth the risk to the 10-day sprint timeline.

IMPACT

Saves 1.5 days. Repetition signal already captures Reddit-style "volume of independent coverage" implicitly. V1.1 roadmap item. Final weights: repetition 70%, Google Trends 30%.

ADR-013

Neural embeddings: curl-based download, TF-IDF safety fallback

Accepted
WHY

HuggingFace model downloads (87MB) use the XetHub CDN. Python's HTTP client times out on this CDN through the Walmart corporate proxy. curl handles chunked downloads correctly with a 300-second timeout. Model downloads at ~790KB/s and completes in ~105 seconds.

IMPACT

Neural model is active and working. TF-IDF fallback remains for cold-start environments (new developer cloning repo without the model). Discrimination ratio 18.3× confirms model quality. Zero ongoing impact — one-time setup step.

ADR-014

feed_diversity replaces pytrends on Walmart network

Accepted
WHY

pytrends 4.9.x returns HTTP 400 due to Google's 2024 auth API change. Fails without proxy too — it's a library issue, not network. All other external APIs (HN, Algolia, Reddit) return 407 (proxy authentication required for non-Google domains). Only Google-owned domains are whitelisted.

IMPACT

Social signal runs offline via editorial breadth: (unique feeds covering cluster - 1) / 4. A story that surfaces in Economy + Politics + Top Stories simultaneously gets a strong signal. pytrends remains in the code and auto-activates in production if Google fixes the API. signal_source field on every scored cluster shows which path ran.

ADR-002

Background pipeline + SQLite cache — never compute on request

Accepted
WHY

The full pipeline (fetch → cluster → score → 5× LLM calls) takes 45–90 seconds minimum. The NFR requires <1 second API response. These two constraints are irreconcilable without a cache layer. Hourly refresh is standard practice for news intelligence products (Google News, Morning Brew).

IMPACT

API always serves from SQLite — response in <500ms. "Last updated X minutes ago" shown in UI. Stale fallback: if pipeline fails, last good brief is served with is_stale: true flag. Pull-to-refresh triggers async refresh but immediately returns cache — user never waits.

Open Questions & Risks

⚠️
LLM output quality — hallucination rate unknown until M3

We have hallucination guardrails designed but not tested. The 80% target for "Pydantic validation passes without retry" is an assumption. If hallucination rate is higher, M3 will take longer to pass. Mitigation: temperature=0.3, source-grounded prompts, structured JSON output, 2× retry on schema failure.

⚠️
Social signal is currently single-dimensional

With feed_diversity as fallback, the social score is 0.0 for all single-feed clusters (most clusters today). Trend ranking is 100% driven by repetition on these days. This is technically correct but means ties are frequent and M3 LLM output becomes the effective tiebreaker — not a scoring metric. V1.1 mitigation: Reddit PRAW or a working pytrends version.

💡
Persona calibration — "Why it matters" needs validation

We've defined the target persona (28–45, Silicon Valley, investor/builder mindset) in the system prompt. But we haven't validated that the LLM actually writes for this persona vs a generic professional. Plan: Build a small red-team eval in M3 QA — 3 human reviewers from target persona rate "relevance to me" on a 1–5 scale.

✅
All infrastructure risks resolved for M1 + M2

Neural model download (Walmart network), API authentication (proxy restrictions), Google Trends auth (pytrends library bug), article deduplication — all solved with fallback paths that are transparent and logged. No open blockers on current milestone.
